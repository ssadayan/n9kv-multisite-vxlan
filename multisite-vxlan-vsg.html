Problem Statement

Single-Site EVPN Doesn't Cross DCs

  • Intra-site VXLAN EVPN works perfectly — VTEP-to-VTEP tunnels, iBGP EVPN control plane, ARP suppression
  • The problem: each site has its own iBGP domain; routes and VTEPs don't cross site boundaries natively
  • Multisite solution: introduce a Border Gateway (BGW) at each site edge
  • BGW re-originates EVPN Type-2/Type-5 routes between sites via inter-site eBGP
  • BGW re-encapsulates VXLAN traffic — inner VTEP stays local, outer VTEP = BGW multisite loopback
  • Key NX-OS command: evpn multisite border-gateway <site-id>

Layer 3 Underlay

Underlay: OSPF

  • Process name: UNDERLAY, area 0.0.0.0
  • All fabric links: ip ospf network point-to-point
  • MTU 9150 on all p2p links (VXLAN overhead)
  • loopback0 (RID) → in OSPF area 0
  • loopback1 (NVE VTEP) → in OSPF area 0
  • loopback100 (BGW multisite) → NOT in OSPF, reached via static route
  • DCI link → NOT in OSPF — isolated from site underlay
! On every fabric node:
router ospf UNDERLAY
  router-id 10.1.0.11

interface loopback0
  ip router ospf UNDERLAY area 0.0.0.0

interface loopback1
  ip router ospf UNDERLAY area 0.0.0.0

interface Ethernet1/1
  mtu 9150
  ip ospf network point-to-point
  ip router ospf UNDERLAY area 0.0.0.0

Layer 2 Overlay

Overlay: iBGP EVPN per Site

  • Each site is a self-contained iBGP EVPN domain
  • BGWs = Route Reflectors (no spine layer): route-reflector-client + retain route-target all
  • Leaves = RR clients, peer with both BGWs via lo0 for redundancy
  • VLAN 10 → VNI 100010, RT 1:100010 shared across both sites
  • NVE source: loopback1 (VTEP IP in OSPF)
  • ARP suppression + BGP ingress replication on all VTEPs
! Leaf EVPN config:
feature nv overlay
nv overlay evpn

vlan 10
  vn-segment 100010

interface nve1
  source-interface loopback1
  member vni 100010
    ingress-replication protocol bgp
    suppress-arp

evpn
  vni 100010 l2
    rd auto
    route-target import 1:100010
    route-target export 1:100010

Key Component

The Border Gateway — What It Does

  • Intra-site role: RR client — participates in site-1 iBGP EVPN like a leaf
  • Inter-site role: eBGP EVPN speaker — peers with remote BGW via loopback100
  • Re-origination: Type-2 MAC/IP routes learned from site-1 leaves are re-advertised to site-2
  • Re-encapsulation: inter-site VXLAN traffic uses BGW loopback100 as outer VTEP, not leaf loopback
  • Fabric tracking: BGW monitors spine connectivity — stops advertising if fabric goes down
  • DCI tracking: BGW monitors DCI link — withdraws inter-site routes if DCI fails

NX-OS Config

BGW Configuration Keys

! Enable multisite (Site-ID unique per site)
evpn multisite border-gateway 1

! Spine-facing (fabric-tracking)
interface Ethernet1/1
  evpn multisite fabric-tracking

! DCI-facing (dci-tracking)
interface Ethernet1/5
  evpn multisite dci-tracking

! NVE — multisite BGW loopback
interface nve1
  multisite border-gateway interface loopback100

! BGP — advertise PIP
router bgp 65001
  address-family l2vpn evpn
    advertise-pip
  • border-gateway 1 — site-ID, unique per site domain
  • fabric-tracking — on all spine-facing interfaces
  • dci-tracking — on DCI interface only
  • loopback100 — multisite VTEP, not in OSPF
  • Reachability via static route: ip route 10.2.100.21/32 172.16.0.2
  • advertise-pip — enables per-VTEP MAC advertisement for cross-site ARP suppression

Inter-Site Control Plane

Inter-Site eBGP Session

! s1-bgw1 — peer to s2-bgw1
ip route 10.2.100.21/32 172.16.0.2

router bgp 65001
  neighbor 10.2.100.21
    remote-as 65002
    update-source loopback100
    ebgp-multihop 5
    address-family l2vpn evpn
      send-community extended

! s2-bgw1 — mirror config
ip route 10.1.100.21/32 172.16.0.1
neighbor 10.1.100.21
  remote-as 65001
  update-source loopback100
  • Peering on loopback100 addresses — not DCI link IPs
  • Static routes provide loopback100 reachability via DCI: 10.x.100.21 → 172.16.0.x
  • ebgp-multihop 5 — loopbacks aren't directly connected
  • update-source loopback100 — BGP source must match the neighbor address
  • Same RT 1:100010 on both sites — routes accepted without rewriting
  • EVPN Type-2 routes cross sites carrying MAC + IP bindings

topology:
  kinds:
    cisco_n9kv:
      image: vrnetlab/cisco_n9kv:9300v-9.3.16

      image: ghcr.io/hellt/network-multitool

    s1-bgw1:
      kind: cisco_n9kv

    s1-bgw2:
      kind: cisco_n9kv
      startup-config: configs/s1-bgw2.cfg
        # s1-leaf1, s1-leaf2 | s2-bgw1, s2-bgw2, s2-leaf1, s2-leaf2
    srv1:
      kind: linux
      exec:
        - ip addr add 192.168.10.1/24 dev eth1
  links:
        # Site 1 — leaf-to-BGW (no spine)
    - endpoints: ["s1-bgw1:eth1", "s1-leaf1:eth1"]
    - endpoints: ["s1-bgw2:eth1", "s1-leaf1:eth2"]

    - endpoints: ["s1-bgw1:eth5", "s2-bgw1:eth5"]
    - endpoints: ["s1-bgw2:eth5", "s2-bgw2:eth5"]

  • Destroy: clab destroy -t multisite-vxlan.clab.yml
  • eth1/eth2 → Ethernet1/1, Ethernet1/2 (leaf uplinks to both BGWs)
  • eth5 → Ethernet1/5 (DCI — bgw1↔bgw1, bgw2↔bgw2)

Packet Walk

Cross-Site Traffic Flow — srv1 → srv3

1
ARP: srv1 (192.168.10.1) ARPs for srv3 (192.168.10.3) — unknown MAC
2
EVPN Type-2: s1-leaf1 learns srv3's MAC/IP via BGP from s1-bgw1, which received it from s2-bgw1
3
ARP suppression: s1-leaf1 proxies ARP reply using the cached MAC — no flood across VXLAN
4
VXLAN encap (intra-site): s1-leaf1 encapsulates frame → outer src VTEP = 10.1.0.111, dst = 10.1.100.21 (s1-bgw1)
5
BGW re-encap: s1-bgw1 strips outer header, re-encapsulates → outer src = 10.1.100.21, dst = 10.2.100.21 (s2-bgw1)
6
Site 2 delivery: s2-bgw1 decaps, re-encaps to s2-leaf1 (10.2.0.111), which delivers to srv3

Architecture

Key Design Decisions

Spineless — BGWs as RRs

BGWs serve dual roles: iBGP EVPN Route Reflectors for leaves and multisite border gateways. Eliminates the spine layer, reducing node count from 11 to 8 and saving ~20GB RAM in the lab.

Dual DCI Links

Two independent DCI paths — bgw1↔bgw1 (172.16.0.0/30) and bgw2↔bgw2 (172.16.1.0/30). Provides DCI redundancy; each BGW independently tracks its own DCI link.

Shared Route Target

Both sites use 1:100010 instead of auto RT — avoids rewrite-evpn-rt-asn complexity and ensures routes are accepted cross-site without transformation.

MTU 9150 + Dedicated loopback100

VXLAN overhead needs MTU ≥ 9050; 9150 adds headroom. BGW loopback100 (multisite VTEP) is kept out of OSPF — reachable only via static route over DCI, isolating multisite identity from underlay.
